## 動作拆分與判斷所需條件/資料

### 1️⃣ 動作拆分粒度 (SOP 步驟 → METHOD STEP)

| 判斷條件           | 所需資料       | 來源                                 | 範例                                     |
|--------------------|----------------|--------------------------------------|------------------------------------------|
| **動作不可中斷性** | 動作動詞清單   | `ddm_structure.csv` - 語法邏輯       | 「拿取→對準→插入→確認」為 4 個不可中斷動作 |
| **手部切換**       | 左/右/雙手定義 | `ddm_structure.csv` - 手套使用的定義 | 左手保持 + 右手操作 = 拆成 2 步          |
| **目標物變化**     | Key Parts 清單 | `ddm_structure.csv` - 目标物         | 拿取 A → 放下 A → 拿取 B = 拆成 3 步     |
| **位置變化**       | 從哪裡/到哪裡  | `ddm_structure.csv` - 從哪裡/到哪裡  | 從料架→到工作台→到治具 = 拆成 2 步       |
| **工具使用**       | 工具清單       | SOP - 工具/治具欄                    | 拿起電動起子 + 鎖附 + 放下 = 拆成 3 步   |
| **重複次數**       | Freq. 頻率     | SOP 描述 (如「鎖附4顆螺絲」)           | 4 顆螺絲 → 拆成「滑出+鎖附」×4 + 放回      |

#### 拆分規則對照

| 情境                       | 拆分方式    | METHOD STEP 數 |
|----------------------------|-------------|----------------|
| 單一動作 (拿取物品)        | 不拆分      | 1              |
| 拿取 + 放置                | 拆成 2 步   | 2              |
| 拿取 + 去除包裝 + 放置     | 拆成 3 步   | 3              |
| 拿取工具 + 操作 × N + 放回 | 拆成 2+N 步 | 2+N            |
| 拿取 + 對準 + 插入 + 確認  | 拆成 3~4 步 | 3~4            |

### 3️⃣ SIMO 同時動作判斷

| 判斷條件         | 所需資料     | 來源                                 | 判斷規則                 |
|------------------|--------------|--------------------------------------|--------------------------|
| **雙手獨立性**   | 手部使用定義 | `ddm_structure.csv` - 手套使用的定義 | 左右手可同時操作不同物件 |
| **動作互不干擾** | 動作類型     | 語法邏輯                             | 兩動作物理上不衝突       |
| **視線需求**     | 目視確認欄   | I 參數                               | I=0 可不看，I>0 需視線    |
| **空間重疊**     | Layout       | 站點配置圖                           | 兩位置不重疊             |
| **時間重疊**     | 動作時間     | TMU 計算                             | 較短動作可與較長動作同時 |

#### SIMO = Y 的條件 (需全部滿足)

```
┌─────────────────────────────────────────────────────────────┐
│  SIMO = Y 的條件:                                           │
│                                                             │
│  1. 左右手操作 (LH + RH 或 BH 的子動作)                      │
│  2. 動作物理上不衝突 (不同物件/不同位置)                      │
│  3. 至少一個動作 I = 0 (不需目視確認)                        │
│  4. 兩動作可在同一時間內完成                                 │
└─────────────────────────────────────────────────────────────┘
```

#### SIMO 判斷流程圖

```
開始判斷
    │
    ▼
┌─────────────────┐
│ 是否左右手分開? │
└────────┬────────┘
         │
    ┌────┴────┐
    │ 是      │ 否 → SIMO = N
    ▼         │
┌─────────────────┐
│ 是否操作不同物件?│
└────────┬────────┘
         │
    ┌────┴────┐
    │ 是      │ 否 → SIMO = N
    ▼         │
┌─────────────────┐
│ 是否有一方I=0?  │
└────────┬────────┘
         │
    ┌────┴────┐
    │ 是      │ 否 → SIMO = N (都需目視)
    ▼         │
  SIMO = Y    │
```

| 動作組合                | SIMO | 原因          |
|-------------------------|------|---------------|
| LH 保持住 + RH 操作     | Y    | 左手不需視線  |
| LH 抓包裝袋 + RH 拿物件 | Y    | 兩動作可同時  |
| BH 雙手對準插入         | N    | 需協調，不可拆 |
| RH 鎖附 + LH 等待       | N    | 只有單手動作  |
| LH 丟垃圾 + RH 拿下一個 | Y    | 位置不衝突    |

### 2\. 影片資料在「自動化生成」中的價值分析

影片是**動態的時間與空間數據**，它能補足靜態 SOP (PDF/CSV) 無法提供的「隱性知識」。

#### (1) 影片可以補足哪些關鍵資料？ (解決 "Gap")

| **真實動作拆解 (Granularity)** | SOP 通常只寫「組裝 DIMM」，但影片顯示這是由「伸手、抓取、移動、對準、下壓」組成的連續動作。影片能告訴 AI **一個步驟該拆成幾行 CSV**。       | `METHOD STEP`, `SEQUENCE MODELS`                                         |
| **空間距離估算 (Distance)**    | 透過影片中與「人手長度」或「DIMM 長度」的比例，可以直接估算手部移動距離，而不需要去現場量測。                                           | **參數 A (Action Distance)** <br> (例: 伸手距離約等於 2 個 DIMM 長 = A6) |

#### (2) 僅有影片還缺乏什麼？ (Missing Data)

即使有了影片，要生成完美的 MiniMOST 分析表，仍缺以下資訊：

1.  **精確的物理尺寸 (Scale/Reference)**：
      * 影片是 2D 的，且有透視變形。雖然可以估算，但缺乏精確的「公分」數據。
      * **解法**：需要在工作台上放置**比例尺 (Reference Marker)**，或已知尺寸的標準物 (如切割墊上的格線)。
2.  **力回饋/觸覺數據 (Force/Tactile)**：
      * 影片看不出「重量」或「卡扣到位的喀噠聲」。AI 無法判斷是否需要用很大力氣 (影響參數 M 或 X)。
      * **解法**：依賴 SOP 中的文字描述 (如 "用力按壓") 或建立元件資料庫 (DIMM 插槽通常定義為重作業)。
3.  **視線外/遮擋區域 (Occlusion)**：
      * 影片中作業員的身體或手可能會擋住操作點（例如擋住螺絲孔）。
      * **解法**：需要多角度攝影機，或結合 3D CAD 模型進行推演。
4.  **隱形操作 (Invisible Actions)**：
      * 例如「檢查」(Inspect)。作業員盯著看 2 秒，影片看起來是靜止的，AI 可能會以為是發呆。
      * **解法**：需結合 SOP 中的「目視檢查」關鍵字，強制插入 `I (Inspect)` 參數。

### 3\. 自動化生成 Pipeline 設計 (SOP + Video → CSV)

若要開發一套系統，自動從 SOP 和影片生成 MiniMOST CSV，建議的 Pipeline 架構如下：

#### 階段一：資料攝取與前處理 (Data Ingestion)

  * **Input**:
      * SOP 文件 (PDF/Image) -\> `OCR & Layout Analysis`
      * 作業影片 (MP4) -\> `Frame Extraction`
      * **Syntax Library (你的 ddm\_structure.csv)**：這是核心規則庫。
  * **Process**:
      * 將 SOP 文字轉換為結構化的 `Sub_Activities` 清單 (如你目前的 CSV)。
      * 將影片分割為對應這些 Sub\_Activities 的片段 (Video Segmentation)。

#### 階段二：雙軌分析引擎 (Dual-Stream Analysis)

這裡需要兩個 AI 模型並行工作，最後融合結果：

**軌道 A：視覺動作分析 (Vision Stream)**

  * **技術棧**: YOLO (物件偵測) + MediaPipe/OpenPose (骨架/手勢識別) + Depth Estimation (距離估算)。
  * **任務**:
    1.  **Detect**: 識別手 (LH/RH)、工具 (風槍)、物件 (DIMM)。
    2.  **Track**: 追蹤手部軌跡，計算移動距離 (Pixel -\> cm 轉換)。
    3.  **Action Recognition**: 識別動作類型 (抓取 G、移動 M、放置 P、按壓 X)。
    4.  **Timing**: 記錄每個動作的 Frame 數。

**軌道 B：語義邏輯映射 (Logic Stream)**

  * **技術棧**: LLM (GPT-4o / Claude 3.5 Sonnet) + RAG (檢索 Syntax Library)。
  * **任務**:
    1.  理解 SOP 文字 (e.g., "鎖附螺絲")。
    2.  從 Library 查找標準語法 (e.g., "拿取工具...鎖附...放回")。
    3.  預填標準參數 (e.g., 鎖螺絲 X=6 TMU)。

#### 階段三：多模態融合與參數計算 (Fusion & Calculation)

  * **Core Logic**:
      * **校準 A 參數**: 視覺軌道算出的距離 (e.g., 40cm) -\> 查表歸類為 `A16` (30-60cm)。
      * **校準 SIMO**: 視覺軌道顯示雙手同時在動 -\> 設定 `SIMO = Y`。
      * **校準 Freq**: 視覺軌道偵測到重複動作 16 次 -\> 設定 `Freq = 16`。
  * **Output**: 生成初步的 MiniMOST CSV 行。

#### 階段四：人機協作驗證 (Human-in-the-loop)

  * 系統標記出「低信心度」的區域 (例如：被遮擋的動作)。
  * IE 工程師在介面上快速確認或修正參數。

#### 開發準備清單 (Development Roadmap)

1.  **建立標註數據集 (Dataset)**：
      * 需要約 50-100 個類似的「電子組裝動作」影片片段。
      * 標記關鍵幀 (Keyframes)：開始抓取、接觸物件、釋放物件。
2.  **校正環境搭建**：
      * 在拍攝工作台設置標準格線 (Grid) 或已知尺寸標記 (ArUco markers)，用於訓練距離估算模型。
3.  **微調 LLM Prompt**：
      * 設計 Prompt 讓 LLM 能精確讀懂 `ddm_structure.csv` 的規則，並能根據視覺描述輸出 MOST 代碼。

這個 Pipeline 建置完成後，影片就不只是「參考資料」，而是直接驅動工時計算的「感測器數據」，能大幅減少 IE 工程師手動 Key-in 和量測的時間。

### 六、資料驗證規則 (Validation Rules Implementation)

這部分是 Pipeline 的「守門員」，請工程師將此邏輯寫入 Code 中。

1.  **VAL-01 (Cycle Completeness)**:

      * `SELECT count(*) FROM Video_GT WHERE cycle_group_id = 'CG-XX'`
      * 若 Count \!= 3 (Get + Align + Press)，標記 **Data Incomplete**。

2.  **VAL-04 (Freq Consistency)**:

      * 取得 Video Cycle Count: `SELECT count(DISTINCT cycle_group_id)` → 16
      * 取得 Config Freq: 查詢 `Config_Lookup` 表 → 16 (若為 Half) 或 32 (Full)
      * 若 Video Count \!= Config Freq，自動生成 **D-003 (Config Diff)**。

3.  **VAL-06 (Confidence Check)**:

      * 若 `confidence < 0.7` AND `review_status != 'Approved'`
      * 自動將 `review_status` 更新為 **Review Req**。
