### Phase 1: Reference (REF) 設計優點整理

從圖片描述與功能邏輯來看，REF 系統的優勢在於**將「工程師的思維過程」視覺化與結構化**，而不僅僅是處理數據計算。

| 優點面向 | 具體描述 (REF 設計) | 帶來的價值 |
| :--- | :--- | :--- |
| **作業直覺性** | **引導式流程**：先選動作類型 -> 點選參數 -> 選擇補充內容。 | 降低新手門檻，減少誤填代碼的機率，讓操作像是在描述動作而非填寫 Excel。 |
| **動作邏輯性** | **物件導向思維**：透過「目標物、元件、從哪裡、到哪裡」來構建動作。 | 符合現場觀察邏輯（人是去拿東西，不是去移動幾公分），自動關聯手套、工具等屬性。 |
| **雙手協調性** | **組合區概念**：單一動作分析完後暫存在組合區，再依照「用手原則」組合成 MI 語句。 | 完美解決「左右手同時動作 (SIMO)」與「動作平衡」的問題，这是精確工時分析的核心。 |
| **可讀性** | **MI 語句生成**：系統自動生成可讀的自然語言（MI 語句），而非僅顯示代碼。 | 讓非 IE 專業的主管或現場領班也能輕鬆理解 SOP 內容，便於審核與溝通。 |
| **數據完整性** | **補充內容結構化**：強制分類資料（如分類目標物），而非單純文字輸入。 | 確保後續數據分析（如：多少动作是針對螺絲、多少动作需要弯腰）的一致性與準確度。 |

### Phase 2: DDM 關鍵缺乏的具體設計方案

針對目前的後端程式碼 (main.py) 與前端邏輯，以下是填補四大 GAP 的具體設計建議：

#### 1. 缺乏「左右手分配與並行動作 (Simultaneous Motion)」機制

**核心問題：** 目前 `SOPAction` 是線性列表，無法表達「左手拿 PCB，同時右手拿螺絲」的場景。
**後端修改 (main.py)：**
* 修改 `MOSTStep` 或新增 `CompositeStep` 模型，引入 **Hand Assignment** 概念。
* 新增 **SIMO (Simultaneous Motion)** 計算邏輯：取左右手動作中 TMU 較大者作為該步驟的有效時間。

```python
# 設計建議: 新增並行動作結構
class SimoGroup(BaseModel):
    left_hand_action_id: Optional[str] = None
    right_hand_action_id: Optional[str] = None
    description: str
    # SIMO 規則: 一隻手動作時，另一隻手可能是 保持/等待(Delay)/或同時動作
    effective_tmu: int  # 自動計算 MAX(Left.TMU, Right.TMU) 或根據複雜係數調整
```

**前端 UI 設計：**
* **雙軌時間軸**：編輯區分為「左手軌」與「右手軌」。
* **同步標記**：允許用戶將兩個動作（如 Act-001 和 Act-002）選取並點擊「合併/同步」，系統將它們綁定在同一行，視覺上顯示並行。

#### 2. 缺乏「暫存與拖拉組合」的工作流

**核心問題：** 目前是「新增 -> 儲存至列表」，一旦加入就很難拆解或重組。
**後端修改 (main.py)：**
* 新增 **Clipboard / Draft API**：允許暫存未歸檔的動作片段。
* 新增 **Reorder API**：支援拖拉排序（目前有了 `reassign_action` 但僅限跨站點，需要站點內的排序）。

**前端 UI 設計：**
* **工作檯 (Draft) vs. 正式區 (Sequence)**：
    * 左側為「動作積木區」（已分析好的單一動作）。
    * 右側為「SOP 組合區」。
* **拖曳操作**：用戶先在左側快速建立 5 個動作（不分順序），然後拖曳到右側組合區，決定哪些是左手、哪些是右手、哪些先做。

#### 3. 缺乏「自然語言生成 (NLG)」

**核心問題：** `generate_chinese_sentence` 目前僅做簡單字串串接，缺乏流暢度。
**後端修改 (main.py)：**
* 優化 `generate_chinese_sentence`，引入 **動作動詞庫 (Verb Templates)**。
* 根據 `OBJECT_LIBRARY` 的屬性自動潤飾語句。

```python
# 設計建議: 優化後的 NLG 邏輯
def generate_smart_sentence(action, obj, from_loc, to_loc, hand):
    # 範例邏輯:
    # IF action="G"(抓取) AND obj="screw"(螺絲) -> "抓取螺絲"
    # IF action="P"(放置) AND to_loc="hole"(孔) AND modifier="insert" -> "對準並鎖入螺孔"
    template = "{hand} 自 {from_loc} {verb} {object}，{motion_verb} 至 {to_loc}"
    # ... (使用 Python jinja2 或 f-string 進行更複雜的語法組合)
```

**前端 UI 設計：**
* **即時預覽**：當用戶調整參數（例如距離從 30cm 改為 60cm）時，句子自動從「伸手抓取」變為「跨步伸手抓取」，讓用戶確認參數是否符合真實情境。

#### 4. 缺乏「視覺化補充內容介面」

**核心問題：** 目前依賴純文字 ID 或下拉選單，選取「物料」或「工具」時不直觀。
**後端修改 (main.py)：**
* 擴充 `OBJECT_LIBRARY` 與 `TOOL_LIBRARY`，增加 `image_url` 與 `icon` 欄位。
* 增加 **Tag/Category API**：支援前端用「螺絲類」、「線材類」、「外殼類」過濾物件。

```python
# 資料結構擴充
OBJECT_LIBRARY = [
    {
        "id": "obj-screw-01", 
        "name": "M3 螺絲", 
        "category": "Fastener", 
        "image_url": "/static/img/screw_m3.png", # 新增
        "tags": ["metal", "small_part"]           # 新增
    },
    ...
]
```

**前端 UI 設計：**
* **圖示化選擇器 (Icon Picker)**：
    * 選擇「目標物」時，彈出 Grid 視窗顯示物料照片。
    * 選擇「動作」時，顯示 MOST 動作的示意圖（例如：顯示一隻手抓東西的 icon 代表 G）。
* **情境篩選**：如果動作選了「鎖附」，目標物清單自動過濾只顯示「電動起子」或「螺絲」，屏蔽無關選項。
